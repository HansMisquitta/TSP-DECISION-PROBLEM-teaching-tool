NP-Completeness Proof

Traveling Salesman Problem (Decision Version)  |  Reduction from Hamiltonian Cycle

NP-COMPLETE CLO-5 · 20 MARKS

Problem Definition

Formal Statement

TSP-D = { ⟨G, w, B⟩ : G is a complete weighted graph, w assigns positive integer weights to edges, and there exists a Hamiltonian cycle in G with total weight ≤ B }

Input: Complete graph G = (V, E), weight function w: E → ℤ⁺, bound B ∈ ℤ⁺ Question: Does G have a tour visiting all |V| = n vertices with total weight ≤ B?

Step 1 — TSP-D is in NP

Certificate + Polynomial Verifier

Certificate: A permutation π = (v₁, v₂, …, vₙ) of the n cities, representing the claimed tour.

Verifier algorithm (runs in polynomial time):

VERIFY(⟨G, w, B⟩, π): 1. Check |π| = n and all vertices appear exactly once → O(n) 2. Compute total = w(vₙ, v₁) + Σᵢ₌₁ⁿ⁻¹ w(vᵢ, vᵢ₊₁) → O(n) 3. Return (total ≤ B) → O(1) Total verifier runtime: O(n) ⊆ poly(|⟨G,w,B⟩|) ✓
Conclusion: Since a valid certificate can be verified in polynomial time, TSP-D ∈ NP. ✓

Step 2 — NP-Hardness via Reduction from Hamiltonian Cycle

Reduction Construction

We show HAM-CYCLE ≤ₚ TSP-D. Given an arbitrary instance ⟨G⟩ of Hamiltonian Cycle (undirected graph G = (V, E)), construct a TSP-D instance as follows:

Construction f(⟨G⟩) = ⟨G', w, n⟩: G' = complete graph on V (same vertex set) For each pair of vertices u, v ∈ V: w(u, v) = 1 if {u,v} ∈ E (edge exists in G) w(u, v) = 2 if {u,v} ∉ E (edge absent in G) Bound B = n (where n = |V|)

This construction runs in O(n²) time (examine each pair of vertices once) — polynomial in |G|. ✓

Forward Direction (→): G has Ham. Cycle ⟹ G' has tour of weight ≤ n

Suppose G has a Hamiltonian cycle C = (v₁, v₂, …, vₙ, v₁). Then:

• C visits all n vertices exactly once • Every edge {vᵢ, vᵢ₊₁} and {vₙ, v₁} is in E (by definition of Ham. Cycle) • By our weight function, each such edge has weight w = 1 • Total weight = n × 1 = n ≤ B = n ✓
Therefore: The same tour C is a valid TSP-D tour of weight exactly n ≤ B. ✓
Backward Direction (←): G' has tour of weight ≤ n ⟹ G has Ham. Cycle

Suppose G' has a tour T with total weight ≤ n. Then:

• T visits all n vertices → T has exactly n edges • Each edge has weight ≥ 1 (minimum possible) • Total weight ≥ n × 1 = n • But we assumed total weight ≤ n • Therefore total weight = n exactly • This means every edge in T must have weight exactly 1 • By construction, weight 1 ⟺ edge exists in G • Therefore all n edges of T are in E
Therefore: Tour T is a Hamiltonian cycle in the original graph G. ✓
Conclusion
• TSP-D ∈ NP (Step 1) ✓ • HAM-CYCLE ≤ₚ TSP-D (polynomial-time reduction) (Step 2) ✓ • HAM-CYCLE is NP-complete (known result) • Therefore TSP-D is NP-complete (by definition) ✓

NP vs P — Context

TSP is one of the most studied NP-complete problems. The decision version is in NP (cert. verifiable in poly time) but no polynomial-time exact algorithm is known. The optimization version (find minimum tour) is NP-hard. The best exact algorithms run in O(n² · 2ⁿ) via dynamic programming (Held-Karp, 1962). For practical instances, heuristics like nearest-neighbor and 2-opt are used.

Complexity class hierarchy (assuming P ≠ NP): P ⊊ NP ⊇ NP-complete TSP-D ∈ NP-complete If P = NP, then TSP-D ∈ P (polynomial exact solution exists)

Growth Rate Analysis

Brute Force O(n!) vs Nearest-Neighbor O(n²) vs 2-opt O(n² · k)

Brute Force — O(n!)

Enumerates all (n-1)!/2 distinct tours. For n=12, that's 19,958,400 tours. Exact but exponential — infeasible beyond ~15 cities.

n=4: 3 tours n=6: 60 tours n=8: 2,520 tours n=10: 181,440 tours n=12: 19,958,400 tours
Nearest-Neighbor — O(n²)

Greedy heuristic: always go to closest unvisited city. Fast but can produce suboptimal tours — the approximation ratio can be as bad as O(log n) in the worst case.

Worst-case ratio: ≈ ½(⌈log₂n⌉ + 1) Typical practical ratio: 1.2× to 1.5× optimal
NN + 2-Opt — O(n² · k)

Iteratively swaps pairs of edges if the swap reduces tour length. k = number of improvement passes (usually small). Significantly better than pure NN in practice.

2-opt swap: remove edges (i,i+1) and (j,j+1) Add edges (i,j) and (i+1,j+1) Repeat until no improvement found
⚠ Counterexample: When Nearest-Neighbor Fails

Consider 6 cities arranged so that NN greedily picks short local edges but misses a globally better tour. Load the "NN Bad Case" preset in the Visualizer to see this concretely. The NN tour visits cities in an order that backtracks, while the optimal tour uses a shorter overall path.

Bad NN Example: cities at (0,0),(1,0),(2,0),(3,0),(4,0),(2,3) NN from city 0: 0→1→2→3→4→5→0 (may be suboptimal) The NN can produce tours up to 25% worse than optimal on structured inputs.

Educational Overview

A comprehensive guide to the Traveling Salesman Problem, NP-completeness, and algorithm design.

What is TSP? ▼

The Traveling Salesman Problem is one of the most famous problems in computer science and operations research. A salesman must visit n cities exactly once and return to the starting city, minimizing total travel distance (or cost).

Decision version: Given cities, edge weights, and a bound B — does there exist a tour of total weight ≤ B? (YES/NO answer)

Optimization version: Find the minimum weight tour. This is NP-hard.

Real-world applications: Package delivery routing (UPS, FedEx), circuit board drilling, DNA sequencing, telescope scheduling, microchip fabrication, vehicle routing.

What is NP-Completeness? ▼

P = problems solvable in polynomial time. NP = problems where solutions are verifiable in polynomial time. NP-complete = hardest problems in NP; every NP problem reduces to them.

A problem X is NP-complete if: (1) X ∈ NP, and (2) every NP problem reduces to X in polynomial time. Equivalently, it suffices to show reduction from any known NP-complete problem.

If any NP-complete problem is in P, then P = NP (all NP problems are in P). This is the famous P vs NP open problem.

Nearest-Neighbor Heuristic ▼

Design paradigm: Greedy

Algorithm: Start at any city. Repeatedly visit the nearest unvisited city. Return to start when all cities are visited.

NN Algorithm: tour = [start] visited = {start} while |tour| < n: next = argmin{ dist(current, v) : v ∉ visited } tour.append(next) visited.add(next) tour.append(start) // close the cycle

Why it can fail: Greedy choices look locally optimal but ignore global structure. The algorithm can "paint itself into a corner" — taking short edges early forces long edges later.

2-opt Local Search Improvement ▼

Design paradigm: Local Search / Iterative Improvement

After building an initial tour (e.g., via NN), 2-opt iteratively improves it by removing two edges and reconnecting in a different way. If the swap reduces tour length, keep it.

2-opt Improvement: improved = true while improved: improved = false for i in 0..n-2: for j in i+2..n: if dist(tour[i], tour[j]) + dist(tour[i+1], tour[j+1]) < dist(tour[i], tour[i+1]) + dist(tour[j], tour[j+1]): reverse(tour, i+1, j) // reconnect improved = true

Complexity: Each pass is O(n²). In practice, converges in a small number of passes. Typical improvement over NN: 10–20%.

Worked Example — 4 Cities ▼

Cities: A=(0,0), B=(1,0), C=(1,1), D=(0,1)

Decision check (B=4): Does tour of weight ≤ 4 exist? YES (optimal = 4.0)
